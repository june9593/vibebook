## vibebook v0.2 是什么

一个工具，把 user 在多台机器上跟 AI 编程的 raw jsonl session **同步到私有 git repo**
（这是卖点，保持），然后让 user **在 Claude Code 里通过 `/vibebook` slash command**
让当前 in-session Claude 把这些 session 整理成**三种 per-project 产物**：

```
book/<project>/
  chronicle/         # 流水账(给将来的自己 + 周报素材)
    YYYY-MM-DD__<thread-slug>__<tid8>.md
    每篇 = 一个 thread(跨 session,但同一件事;不值得写的 SKIP)
    四段式: What / Why / How / Outcome
  topics/            # 知识库(主题页,跨 session 累积)
    <topic-slug>.md
    每页 = 一个子系统/一类问题(中粒度)
    每个新 thread 涉及到时,LLM 全文重写整页(保留旧 fact + 加新 fact)
  cards/             # 原子卡片(防止重复踩坑;给将来的自己看)
    <slug>.md
    每张 = 一个 atomic insight (gotcha / pattern / decision / howto)
    严格 dedup(LLM 写前先 grep);wikilink 互引
book/_global/cards/  # 跨项目通用卡片(git/macOS/工具使用)
book/index.md        # 三种产物的 catalog 入口
book/_meta/timeline.md  # 全局时间线
```

**核心信念**:
- **Sync 是 user 手动跑的 CLI**(`vibebook sync`),只做 raw 同步,**不调 LLM**
- **写作完全在 user 本地 Claude Code 里**通过 `/vibebook` 触发,由 in-session Claude
  做 — 因为 in-session 才有完整 context、能 Read/Grep 别的 article、能被 user 打断指导
- **质量 > 速度 > token 成本**(user 明确说不在乎后两者)
- **per-project 隔离是硬性规则**:edge-src 的踩坑不能污染 chameleon 的知识库;
  跨项目通用 insight 单独走 `_global/cards/`

## 三种产物的形状(详细)

### 1. Chronicle(流水账) — `book/<project>/chronicle/`

**单位**: 一个 thread(跨 session,但同一件事)。
**SKIP 规则**: 只 say hi、只让 agent 读代码没做事、只问 single short question 而没
后续 — SKIP,不硬写。
**不强制融合**: 不是同一件事的 session 不要硬合 thread。Threading 算法宁可分细
不要 over-merge。
**结构**:
```yaml
---
title: 修复 Edge 全屏书签条 bug
created: 2026-04-22
updated: 2026-04-23
project: edge-src
threadId: fix-fullscreen-bookmark-bar
sessionIds: [abc12345, def67890]
tags: [fullscreen, bookmark-bar, mac]
relatedTopics: [topics/native-ui-fullscreen.md]
relatedCards: [cards/gotcha-immersive-mode-controller-mac.md]
---

# 修复 Edge 全屏书签条 bug

## What — 做了什么
(一段话或几个 bullet 说清楚成果是什么)

## Why — 为什么要做
(背景 + 触发原因;周报需要的 "影响范围"、"严重性")

## How — 怎么做的
(技术路径、关键代码改动、关键决策点;**保留代码片段、命令行**)

## Outcome — 成果如何
(修了多少 bug / 写了多少行 / merged commit / 数字化)

## 附:原始对话
- raw_sessions/claude/edge-src/2026-04-22/.../abc12345.md
```

### 2. Topics(知识库) — `book/<project>/topics/`

**粒度**: 中 — 一个子系统 / 一类问题(`native-ui-fullscreen.md`、`bookmark-bar.md`、
`omnibox-search.md`)。不是一个具体 bug,也不是整个 project。
**写作时机**: 每次 `/vibebook` 跑完一个新 thread,LLM 判断这个 thread 涉及哪些 topic
- 不存在 → 新建 topic page
- 存在 → **全文重写整页**(读旧 page + 加新 fact + 重新组织叙事)
- 不涉及任何 topic(纯 misc) → 不动 topics/
**Full rewrite 不能丢历史**: SKILL.md 里写死规则:LLM 重写时必须读旧 page 全文,
保留所有 historical fact + 加 new fact + 重组结构。失败时备份旧版到 `.bak`。
**结构**:
```yaml
---
title: Edge Native UI 全屏行为
project: edge-src
topic: native-ui-fullscreen
created: 2026-04-15
updated: 2026-04-23
contributingThreads: [fix-fullscreen-bookmark-bar, immersive-mode-rewrite]
relatedCards: [gotcha-immersive-mode-controller-mac.md]
---

# Edge Native UI 全屏行为

## 这个 topic 是什么
(一段总览)

## 关键概念
(子系统的核心概念、文件、抽象)

## 历史问题与修法
(每个 thread 解决过什么 — 按时间倒序;每条引一个 chronicle 链接)

## 当前已知坑
(active gotcha,引用 cards/)

## 相关
- [[chronicle/fix-fullscreen-bookmark-bar]]
- [[cards/gotcha-immersive-mode-controller-mac]]
```

### 3. Cards(原子卡片) — `book/<project>/cards/` + `book/_global/cards/`

**单位**: 一张卡 = **一个 atomic insight**。
**LLM 取舍** (偷 memex 的写作规范,**完整原文塞进 SKILL.md**):
- **Atomic**: 能拆就拆;一张卡一个事
- **Non-obvious test**: "如果不写下来,下次做类似事会失去这个 insight 吗?不会 → 别写"
- **Own words / Feynman**: 用自己的话,不要粘 API 文档原文 / 错误信息原文
- **Fact Hygiene Check**: 写完自检三问 WHO / WHAT-WHEN / RELATIONSHIP (memex 原文)
- **Dedup before write**: 写前 `Glob book/<proj>/cards/*.md` 看有没有相似的;
  有就 update existing 不要 fragment
**Slug 规范**: `gotcha-*` / `pattern-*` / `decision-*` / `howto-*` / `tool-*`
**Per-project vs global 决策规则**(SKILL.md 里写死):
- Topic 跟具体 project / codebase / 业务领域绑定 → `book/<project>/cards/`
- Topic 是工具 / 语言 / OS / 通用 best-practice → `book/_global/cards/`
- 二者都涉及 → 优先 per-project,在 `_global/` 留一个简短 wikilink 入口

**结构**:
```yaml
---
title: ImmersiveModeControllerMac 在子线程被销毁导致 UAF
slug: gotcha-immersive-mode-controller-mac-uaf
type: gotcha
project: edge-src        # 或 _global
created: 2026-04-22
tags: [mac, fullscreen, threading]
---

(一段话讲清这个 fact;包含 wikilink 解释 WHY related,不是裸 "Related: [[x]]")

参考:[[chronicle/fix-fullscreen-bookmark-bar]]
```

## 触发模型

### 入口 1:`vibebook sync`(CLI,user 手动)
**只做 raw 同步**:
1. extract jsonl from `~/.claude/projects/` + `~/Library/Application Support/.../GitHub.copilot-chat/`
2. encrypt(if cfg.encrypt)+ commit `raw_sessions/` + `.vibebook/index.json`
3. fast-forward + push device branch

**不调 LLM**。**不写 book/**。这是质量天花板的物理保证 — sync 永远不依赖 Claude
状态、不依赖 token 预算、不会因为 LLM 慢/挂而失败。

### 入口 2:`/vibebook` slash command(在 Claude Code 里)
**写书全流程**,由 in-session Claude 驱动。`skills/vibebook/SKILL.md` 写明 7 步:

1. **Sync 检查**: 提示 user 是否已跑过 `vibebook sync`(没跑就先跑;给命令)
2. **Plan**: `bash` 调 `vibebook prepare --project <p>` 拿 JSON:
   ```ts
   {
     project: string,
     newThreads: Array<{
       threadId, sessionIds, score, project,
       sessionPreviews: [{shortId, endedAt, preview, insightScore}]
     }>,
     existingTopics: string[],     // 已有 topic slug 列表
     existingCards: string[],      // 已有 card slug 列表
     segmentationPrompt: string    // 嵌入,告诉 LLM 怎么分 thread
   }
   ```
3. **Threading**: in-session LLM 看 `segmentationPrompt` + `newThreads`,**自己**做
   thread 切分,产出 `groups: [{threadId, sessionIds, title, skip:bool, skipReason}]`
   写到 `/tmp/vibebook-groups.json`
4. **Chronicle 写作**: 对每个非 skip group,LLM **inline** 写四段式 chronicle 到
   `/tmp/vibebook-chronicles.json`(数组)。SKILL.md 在这里塞 chronicle 写作规范
   (4 段 + frontmatter)
5. **Topic 更新**: LLM 判断每个 thread 涉及哪些 topic;
   - 涉及已有 topic → `Read book/<p>/topics/<slug>.md` 全文 → 重写 → 写入
     `/tmp/vibebook-topics.json`(标 update)
   - 涉及新 topic → 新建,标 insert
   - 不涉及 → skip
   SKILL.md 塞 topic 写作规范(中粒度 + 不丢历史 + 失败备份 .bak)
6. **Card 提取**: LLM 从每个 chronicle 提取 0-N 张 atomic insight,
   先 `Glob book/<p>/cards/*.md` + `Glob book/_global/cards/*.md` dedup,然后写
   `/tmp/vibebook-cards.json`。SKILL.md 塞 memex 完整规范(atomic / non-obvious /
   Feynman / Fact Hygiene Check / per-project vs global)
7. **Publish**: `bash` 调 `vibebook publish --chronicles ... --topics ... --cards ...`
   - chronicle: append-only insert(threadId 撞了报警)
   - topic: update(读旧 → 备份 .bak → 写新)
   - card: insert + dedup(slug 撞了 update existing)
   - 重新生成 `book/index.md` + `book/_meta/timeline.md`
   - 自动 git add + commit + push device branch

每一步 user 能看到 Claude 在干什么,可以打断、可以追问、可以让它"再补一段细节"。

### 入口 3:`/vibebook regenerate <thread-id>`(可选,Sprint 5 再说)
重新生成某条 chronicle / 重写某个 topic — 也走 in-session。

## 跟旧 plan 的差异(自评)

| 维度 | 旧 plan(机械缝合) | 新 plan(吸收后) |
